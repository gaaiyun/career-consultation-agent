# 实施路线图

## 1. 总体策略

采用“先流程可用，再能力增强，再系统外联”的路线。

原则：

- 第一优先级是把咨询流程跑通
- 第二优先级是提高输出质量和可编辑性
- 第三优先级是打通飞书与数据沉淀

## 2. 版本路线

## 2.1 V0.1 MVP

目标：让咨询师可以真实使用完成单案例全流程。

范围：

- Streamlit 单页应用
- SQLite 本地数据库
- SiliconFlow 接口接入
- 四阶段 Prompt
- 基础案例管理
- Markdown 导出

验收标准：

- 本地可运行
- 能完成 1 个案例从录入到终版报告

## 2.2 V0.2 质量优化版

目标：提升稳定性和可编辑性。

新增内容：

- 历史版本回看
- 更细的错误提示
- 更稳定的 JSON 解析
- Prompt 调优
- 更好的页面布局

## 2.3 V0.3 飞书预集成版

目标：把本地结构化数据同步到飞书多维表格。

新增内容：

- 飞书认证配置
- 单案例手动同步
- 字段映射器
- 同步结果状态记录

## 2.4 V0.4 轻量生产版

目标：支持更高频的真实咨询使用。

新增内容：

- 检索与标签过滤
- 多模板风格切换
- 案例导出管理
- 可选 PostgreSQL

## 3. 技术选型说明

## 3.1 前端与交互

- `Streamlit`

原因：

- 足够支撑咨询师内部工具
- 表单、文本和结果页开发速度快

## 3.2 服务编排

- 本地 Python 服务式组织

原因：

- MVP 阶段不需要引入 FastAPI 等额外服务层
- 降低复杂度

## 3.3 模型调用

- `openai` Python SDK
- `SiliconFlow` OpenAI 兼容接口

原因：

- 后续替换模型供应商成本低

## 3.4 数据存储

- `SQLite`

原因：

- 单人本地工具足够
- 无部署门槛

后续升级：

- `PostgreSQL`

## 3.5 外部集成

- 飞书多维表格 OpenAPI

原因：

- 适合做案例台账、阶段记录与复盘面板

## 4. 迭代里程碑

## M1：项目骨架

- 初始化目录结构
- 配置文件
- LLM Adapter
- SQLite 初始化

## M2：核心工作流

- 结构化拆解
- 追问生成
- 路线规划
- 终版报告

## M3：文档与体验

- PRD
- 架构文档
- Prompt 文档
- 数据模型文档
- 页面体验优化

## M4：飞书同步

- 飞书客户端
- 映射器
- 单案例同步

## 5. 风险控制

### 5.1 模型输出不稳定

控制方式：

- 限制输出结构
- 本地校验
- 分阶段生成

### 5.2 Prompt 成本失控

控制方式：

- 单阶段单次调用
- 保存输出避免重复跑

### 5.3 数据迁移困难

控制方式：

- 一开始就按结构化字段设计
- 保留统一领域模型

## 6. 建议实施顺序

1. 先跑通最小代码骨架
2. 再完善文档和 Prompt
3. 再做页面交互和导出
4. 最后接飞书

## 7. 团队分工建议

如果后续多人协作，建议分工如下：

- 产品/咨询方法：维护 PRD 和 Prompt
- 工程：维护 Streamlit、存储、LLM Adapter
- 运营/咨询师：验证案例效果和报告质量
